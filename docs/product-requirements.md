# Product Requirements

This PRD turns the product brief into buildable first-release requirements.

## Problem Statement

Laravel teams lose time during staging review because technical context is split
across logs, database tools, queues, scheduled task output, browser reports, and
chat messages. Strata should make short-lived staging review behavior visible in
one private dashboard while keeping sensitive data out by default.

## Product Goal

For the first release, a Laravel team should be able to install Strata in a
staging application, open a private dashboard, perform a review workflow, and
understand the technical activity generated by that workflow without exposing
request bodies, cookies, tokens, raw bindings, or job payloads.

## User Stories

- As a developer, I want to see recent staging requests and related telemetry so
  I can quickly identify what happened during a reported issue.
- As a QA tester, I want to share safe technical context with a bug report so a
  developer can investigate without asking me to reproduce repeatedly.
- As a technical lead, I want to scan staging health before a review so I can
  decide whether the environment is ready.
- As a client-service teammate, I want a redacted issue summary so I can support
  client review without handling sensitive technical data.
- As a package adopter, I want clear configuration defaults so I can install
  Strata without accidentally exposing production or private data.

## Functional Requirements

### Foundation

- Document product scope, non-goals, personas, and staging workflows.
- Document telemetry capture categories and redaction policy.
- Document package architecture, storage, retention, compatibility, testing,
  dashboard access, and configuration decisions.
- Maintain a decision log for durable product and architecture decisions.

### Prototype

- Provide an installable Laravel package skeleton.
- Publish a single readable configuration file.
- Register isolated dashboard routes and views.
- Provide a package test harness.
- Prototype a dashboard shell with timeline, filters, empty state, and safe
  event detail placeholders.

### Alpha

- Capture request lifecycle metadata.
- Capture query timing and SQL shape metadata without raw bindings.
- Mark slow queries and cautious possible N+1 signals.
- Capture exception, job, and scheduled task lifecycle metadata.
- Store normalized, redacted telemetry events.
- Render a bounded dashboard timeline from stored events.
- Enforce dashboard access rules.

### Beta

- Harden storage and retention behavior.
- Improve dashboard usability for repeated review workflows.
- Add issue context sharing that preserves redaction and authorization.
- Validate privacy and security assumptions against realistic staging use.
- Expand install, upgrade, troubleshooting, and release documentation.

### 1.0

- Finalize commercial add-on licensing and support policy.
- Publish stable installation and upgrade documentation.
- Verify clean package install in a Laravel 13 app.
- Complete release-blocking test, security, and documentation reviews.
- Tag and publish the stable package.

## Non-Functional Requirements

### Privacy

- Redact or exclude sensitive values before storage.
- Avoid full request bodies, cookies, tokens, raw query bindings, serialized job
  payloads, command output, and environment variables by default.
- Require explicit, narrow opt-in for higher-risk fields.
- Preserve redaction in dashboard views and shared issue context.

### Security

- Keep dashboard routes private by default.
- Disable or strongly guard production usage.
- Use host application middleware and authorization extension points.
- Do not let shared issue links bypass authorization.
- Treat dashboard access, telemetry storage, and redaction as release-blocking
  security surfaces.

### Reliability

- Never break the host application when telemetry capture or storage fails.
- Prefer losing telemetry over exposing sensitive data.
- Show actionable dashboard error states when reads fail.
- Bound retention, dashboard reads, and cleanup behavior.

### Performance

- Keep the first transport in-process and simple.
- Store normalized event metadata, not raw framework payloads.
- Use bounded dashboard reads.
- Revisit transport only when staging workloads show the need for queues,
  polling, server-sent events, or WebSockets.

### Maintainability

- Prefer Laravel public APIs and conventional package structure.
- Keep capture, redaction, storage, and dashboard concerns separable.
- Add abstractions only at real ownership boundaries.
- Keep tests focused on safety contracts and user-observable behavior.

## Release Readiness

The first release is useful when it can support a short staging review session,
surface recent technical activity, protect sensitive data by default, and give
developers enough redacted context to reproduce or triage issues.
