# Product Brief

Strata is a real-time staging telemetry dashboard for Laravel teams.

## Product Boundary

Strata helps teams understand what happened in a staging environment while a
review is happening. It is intentionally focused on short-lived staging review
sessions where QA testers, clients, developers, and technical leads need shared
context.

The first version should answer practical questions:

- What request, query, job, task, or exception just happened?
- Which staging action caused it?
- Is anything obviously slow, repeated, failing, or misconfigured?
- What context can be shared with a developer without exposing sensitive data?
- What should the team inspect next?

Strata should feel like a live staging workbench, not a production monitoring
suite.

## Problem

Staging environments are where QA testers, clients, and developers often find
the issues that did not appear during local development. The problem is that the
technical context is scattered across logs, database tools, queue workers,
browser consoles, and chat messages.

Strata should make staging behavior visible as it happens.

## Audience

The first personas and staging workflows are documented in
[Personas and Staging Workflows](personas-workflows.md).

### Primary Users

- Laravel developers who need fast context when a staging issue appears.
- QA testers who need better evidence than screenshots and chat messages.
- Technical leads who need to understand whether a staging build is healthy
  enough for review.
- Agencies and client-service teams that run frequent client staging reviews.

### Secondary Users

- Product managers reviewing staging behavior before release.
- Client stakeholders validating workflows in a staging environment.
- Support or operations teammates helping triage pre-production issues.

Strata is not designed for anonymous public visitors, production incident
responders, or non-technical analytics consumers in its first release.

## Core Use Cases

- Watch requests and errors during QA sessions.
- Identify slow queries and N+1 patterns.
- Confirm jobs and scheduled tasks are running.
- Capture enough context to reproduce a staging issue.
- Share issue context without exposing sensitive data.

## Supported Staging Workflows

The first release should support these workflows well enough to be useful:

1. A QA tester clicks through a staging build while a developer watches live
   request, query, job, task, and exception activity.
2. A client reviews a staging feature and the team captures a short window of
   technical context around a reported issue.
3. A developer reproduces a staging-only bug and uses Strata to compare the
   request timeline, database queries, queued jobs, and exceptions.
4. A technical lead checks whether a staging environment is safe to hand to QA
   or a client before a review session begins.
5. A team reviews slow queries, repeated query patterns, failed jobs, and failed
   scheduled tasks found during a staging pass.

These workflows assume authenticated staging access and a team-controlled
environment. Strata should not assume public production traffic.

## First Release Definition of Useful

Buildable requirements for the first release are documented in
[Product Requirements](product-requirements.md).

Strata is useful for its first release when a Laravel team can install it in a
staging application, open a dashboard, perform a review workflow, and understand
the technical activity generated by that workflow without digging through
multiple tools.

The first release should provide:

- safe installation defaults for staging environments
- a clear dashboard timeline of recent staging activity
- request, query, job, scheduled task, and exception visibility
- redaction-first handling of sensitive data
- enough context to reproduce or triage a staging issue
- documentation that explains setup, safe usage, limitations, and cleanup
- tests that protect capture behavior, redaction behavior, and dashboard access

The release does not need to be exhaustive. It does need to be trustworthy.

## Non-Goals

- Production observability replacement
- Full APM platform
- Log aggregation platform
- Error tracker replacement
- Business analytics dashboard
- Uptime monitoring
- Synthetic monitoring
- Distributed tracing across multiple services
- Long-term metrics warehouse
- Public status page
- Customer analytics or conversion tracking
- Session replay
- Browser performance monitoring beyond staging issue context
- Infrastructure monitoring
- Alerting or incident management system
- Multi-language telemetry platform

Strata can integrate with those tools later, but it should stay focused on
staging feedback first.

## Scope Guardrails

- Favor staging clarity over broad observability coverage.
- Prefer safe defaults over maximum data capture.
- Treat redaction, retention, and dashboard access as core product behavior.
- Keep the dashboard focused on triage and review, not executive reporting.
- Add integrations only when the core staging workflow is already solid.
- Avoid production-first assumptions until the staging product is proven.
